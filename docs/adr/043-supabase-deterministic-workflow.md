# ADR-043: Supabase Deterministic Workflow — Migrations as Single Source of Truth

**Status:** Accepted
**Date:** 2026-03-16

## Context

Supabase provides a Dashboard UI that allows direct database modifications (creating tables, adding columns, changing RLS policies). This is convenient for prototyping but catastrophic for a production workflow:

1. **Dashboard changes are not tracked in git** — no audit trail, no rollback
2. **Local and production drift** — developer adds a column via Dashboard in staging, forgets to create a migration, production doesn't have it
3. **No reproducibility** — `supabase db reset` only applies migration files, so Dashboard changes are lost
4. **AI agents use whatever works** — agents will use Dashboard, raw SQL, `supabase db push`, or migrations interchangeably unless constrained

## Decision

**Migration files are the Single Source of Truth for all database state.** No exceptions.

### Rules

1. **NEVER modify the database via Supabase Dashboard** for structural changes (tables, columns, indexes, RLS, triggers, functions). Dashboard is read-only for inspection.

2. **NEVER use `supabase db push` from developer machines.** Only CI pipelines are authorized to run `supabase db push` against staging/production (it applies migration files to the remote database). Developers must use `supabase db reset` locally.

3. **ALL schema changes go through migration files:**
   ```bash
   # Create a new migration
   supabase migration new <description>

   # Edit the generated file in supabase/migrations/
   # Apply locally
   supabase db reset

   # Verify
   supabase db diff  # should show no drift
   ```

4. **`supabase db reset` must succeed from zero** — this proves your migrations are complete and idempotent. If `db reset` fails, your migrations are broken.

5. **`supabase db diff` must show zero drift** — if there's drift, either a Dashboard change leaked in or a migration is incomplete.

6. **Seed data goes in `supabase/seed.sql`** — not in migrations (migrations are structural, seeds are data).

7. **Local development uses `supabase start`** — this runs a local Postgres + Auth + Storage + Realtime identical to production. Never develop against a remote Supabase instance.

### Migration File Rules (extends ADR-019)

- Naming: `YYYYMMDDHHMMSS_description.sql` (auto-generated by `supabase migration new`)
- Idempotent: use `IF NOT EXISTS`, `IF EXISTS`
- Non-destructive: no `DROP COLUMN`, no `ALTER TYPE` (ADR-019)
- Every `CREATE TABLE` includes `ENABLE ROW LEVEL SECURITY` (ADR-040)
- Check in ALL migration files to git
- Never manually edit timestamps in migration filenames

### Environment Parity

| Environment | Database | How to Set Up |
|-------------|----------|---------------|
| Local dev | `supabase start` → local Postgres | Automatic, runs migrations |
| CI | Supabase CLI + `db reset` | GitHub Actions service |
| Staging | Supabase project (linked) | `supabase db push` from CI only |
| Production | Supabase project (linked) | `supabase db push` from CI only |

**Critical:** `supabase db push` is only allowed from CI pipelines, never from developer machines. This prevents accidental pushes to staging/production.

## Consequences

**Positive:**
- 100% reproducible database state from git history
- `supabase db reset` is the ultimate test — if it passes, migrations are correct
- New developers get a working database in one command
- No drift between local, CI, staging, and production
- Full audit trail in git

**Negative:**
- Slower than clicking buttons in Dashboard
- Requires discipline (Dashboard is tempting)
- `supabase db push` in CI needs auth tokens (infra setup)

**Enforcement:**
- CI step: `supabase db reset` must succeed (validates all migrations from scratch)
- Migration linter (`scripts/lint-migrations.sh`): structural rules (ADR-019, ADR-040)
- `supabase db diff` in CI: zero drift check (once remote is linked)
- CLAUDE.md: explicit "NEVER use Dashboard for structural changes"
- Pre-push hook: run `supabase db reset` locally before pushing migration changes
